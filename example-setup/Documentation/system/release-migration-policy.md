# Release Migration Policy

> Canonical policy for release-aware schema migration, git tag enforcement, fixture store testing, and upgrade path verification.
>
> For the core persistence policy (when migration is required, reset rules, PR expectations), see `Documentation/system/persistence-policy.md`.
>
> For SwiftData migration implementation details (schema history, lightweight vs custom, testing contract), see `Documentation/system/swiftdata-migrations.md`.

**Version:** 1.0
**Status:** Example Template

## Purpose

[YourApp] ships to real users via TestFlight and the App Store. Once a build leaves the building, its schema version is permanently in the wild. Users do not update predictably. Someone may install v1.0.0, ignore the app for six months, then open it at v3.2.0. If the upgrade path from v1.0.0 to v3.2.0 does not work, that user loses their data or hits a crash on launch.

This policy exists to make that scenario impossible by design, not by hope.

---

## Core Principles

- Every App Store and TestFlight release is tagged in git. The tag is the permanent record of what shipped.
- The migration chain is append-only. Released schema versions and their migration stages are frozen forever.
- Every PR that touches persistence must prove migration works from every released schema version to the current HEAD.
- Proof is automated: seeded fixture stores per release tag, opened through the real migration plan in CI.
- No PR merges if any fixture store fails to migrate.

---

## Git Tag Protocol

### Tag Format

```
release/X.Y.Z
```

Where `X.Y.Z` matches the version string submitted to App Store Connect or distributed via TestFlight.

Examples: `release/1.0.0`, `release/1.1.0`, `release/2.0.0`.

### When To Tag

Tag immediately before submitting the archive to App Store Connect or distributing via TestFlight. The tag marks the exact commit that shipped.

### Tag Rules

- Tags are immutable. Never delete or move a release tag.
- Tags must be annotated, not lightweight. Include the schema version in the annotation message.
- Every release tag must have a corresponding entry in the release migration registry.
- Every release tag must have a corresponding fixture store committed to the test bundle.

### Annotated Tag Command

```bash
git tag -a release/X.Y.Z -m "Release X.Y.Z — SchemaVN"
```

### Tag Checklist (Pre-Release)

Before cutting a release tag:

1. Confirm the migration chain is complete and ordered in `PersistenceMigrationPlan`.
2. Confirm fixture stores exist for all prior release tags.
3. Run the full migration test suite against all fixture stores.
4. Generate and commit the fixture store for the new release.
5. Update the release migration registry.
6. Cut the annotated tag.

## Release Migration Registry

A Swift file that maps release tags to schema versions and fixture store filenames. The test harness iterates over this registry.

Location: `[YourApp]Tests/Persistence/ReleaseMigrationRegistry.swift`

```swift
/// Maps each shipped release to its schema version and test fixture.
/// Append-only. Never remove or reorder entries.
enum ReleaseMigrationRegistry {

    struct Release {
        let tag: String
        let schemaVersion: Schema.Version
        let fixtureStoreName: String
    }

    /// Every release that has shipped to TestFlight or the App Store.
    /// Order: chronological. Oldest first.
    static let releases: [Release] = [
        // Release(
        //     tag: "release/1.0.0",
        //     schemaVersion: Schema.Version(1, 0, 0),
        //     fixtureStoreName: "release_1_0_0_schemaV1"
        // ),
    ]
}
```

### Registry Rules

- Append-only. Never remove, reorder, or edit existing entries.
- Every entry must have a matching fixture store in the test bundle.
- The `schemaVersion` must match the `versionIdentifier` of the `VersionedSchema` that was current at that release.
- The registry is the single source of truth for "what schema versions are in the wild."

## Fixture Stores

A fixture store is a real on-disk SQLite database created from a specific `VersionedSchema`, seeded with representative data for every model that existed at that schema version.

### Purpose

Fixture stores prove that the current migration plan can open a store from any previously released schema version, walk the full migration chain, and arrive at the current schema with data intact.

In-memory tests are not sufficient. They do not exercise the real SQLite migration path.

### Location

```
[YourApp]Tests/Persistence/Fixtures/
├── release_1_0_0_schemaV1.store
├── release_1_1_0_schemaV2.store
└── ...
```

### Generation

Fixture stores are generated by a dedicated test helper or script that:

1. Creates a `ModelContainer` using the `VersionedSchema` that was current at the target release.
2. Seeds representative rows for every model in that schema. At minimum: one row per entity, with non-trivial field values that exercise edge cases (optionals populated, relationships linked, boundary values present).
3. Saves the store to disk.
4. Copies the store file into the test fixtures directory.

The generation code itself must be versioned and deterministic. Given the same schema version and seed logic, it must produce the same store structure. Fixture stores should be committed to the repo as binary test resources.

### Seeded Data Requirements

Each fixture store must contain:

- At least one instance of every persisted model that existed at that schema version.
- Non-default values for every non-optional field (do not rely on zero/empty/nil defaults surviving migration — prove real data survives).
- At least one instance with populated optional fields (prove optionals migrate correctly).
- Relationship references that existed at that version (prove referential integrity survives).
- Edge-case values where relevant: empty strings, zero dates, maximum-length strings, boundary numeric values.

The goal is to create a store that is representative enough that if migration corrupts or drops data, the test catches it.

### Fixture Store Rules

- One fixture store per release tag. Named to match the registry entry.
- Fixture stores are immutable after commit. Never regenerate a fixture for an old release.
- Fixture stores are committed as binary test resources, not generated at test time.
- If a fixture store cannot be generated for an old release (because the schema generation code has moved on), that release's fixture must already exist from when the tag was cut.

## Migration Chain Rules

### The Chain Is Append-Only

SwiftData executes migration stages sequentially: V1->V2->V3. There is no skip mechanism. A user on V1 who opens at V3 executes every intermediate stage.

Consequences:

- Never remove a migration stage from `PersistenceMigrationPlan.stages`.
- Never remove a `VersionedSchema` from `PersistenceMigrationPlan.schemas`.
- Never edit a historical `VersionedSchema` definition (frozen snapshot rule from `swiftdata-migrations.md`).
- Every stage that has been included in a release tag is frozen permanently.

### Squashing Unreleased Stages

During development between two release tags, multiple schema versions may accumulate. If none of those intermediate versions have been released (no release tag includes them), they may be squashed into a single version step before the next release tag is cut.

Example: Release `release/1.0.0` shipped with SchemaV1. During development, SchemaV2 and V3 are added. Before cutting `release/1.1.0`, if V2 was never released, it can be collapsed. The migration plan would go V1->V3 directly.

Squashing rules:

- Only squash versions that have never appeared in a release tag.
- After squashing, the fixture store for the previous release must still migrate successfully to the new schema.
- Run the full migration test suite after squashing to prove nothing broke.
- Document the squash in the PR description.

### Why You Cannot Squash Released Stages

If `release/1.1.0` shipped with SchemaV2, and `release/1.2.0` ships with SchemaV3, you cannot collapse V2->V3 into a single step if V1->V2 was already released. A user who is still on 1.0.0 has a V1 store and needs the full chain: V1->V2->V3.

## CI Enforcement

### Migration Test Suite

Location: `[YourApp]Tests/Persistence/ReleaseMigrationTests.swift`

The test suite:

1. Iterates over every entry in `ReleaseMigrationRegistry.releases`.
2. For each entry, copies the corresponding fixture store to a temporary location.
3. Opens the store through `PersistenceContainer` with the real `PersistenceMigrationPlan`.
4. Verifies the store opens without error.
5. Verifies every model type in the current schema can be fetched.
6. Verifies seeded data survived migration (spot-check critical fields, identifiers, relationships).

```swift
final class ReleaseMigrationTests: XCTestCase {

    func testAllReleasedStoresMigrateToCurrentSchema() throws {
        for release in ReleaseMigrationRegistry.releases {
            let fixtureURL = try XCTUnwrap(
                Bundle(for: Self.self).url(
                    forResource: release.fixtureStoreName,
                    withExtension: "store"
                ),
                "Missing fixture store for \(release.tag)"
            )

            // Copy to temp location so the original fixture is not mutated
            let tempDir = FileManager.default.temporaryDirectory
                .appendingPathComponent(UUID().uuidString)
            try FileManager.default.createDirectory(at: tempDir, withIntermediateDirectories: true)
            let tempStoreURL = tempDir.appendingPathComponent("default.store")
            try FileManager.default.copyItem(at: fixtureURL, to: tempStoreURL)

            // Open through the real migration plan
            let config = ModelConfiguration(url: tempStoreURL)
            let container = try ModelContainer(
                for: PersistenceSchema.schema,
                migrationPlan: PersistenceMigrationPlan.self,
                configurations: [config]
            )

            // Verify store opens and models are queryable
            let context = ModelContext(container)
            // ... fetch and verify seeded data per release fixture expectations

            // Cleanup
            try? FileManager.default.removeItem(at: tempDir)
        }
    }
}
```

### CI Pipeline Rule

**The migration test suite must pass for every PR.** If any fixture store fails to open, fails to migrate, or loses data, the PR is blocked. No exceptions.

## PR Requirements (Persistence-Affecting Changes)

Any PR that touches persisted models must also:

1. Confirm migration works from every released fixture store to the current schema.
2. If a new schema version was added, confirm the fixture stores for all prior releases still migrate.
3. If this PR will be tagged as a release, include the new fixture store and registry entry.

### PR Description Addition

For persistence-affecting PRs, add this section:

```markdown
## Release Migration

- [ ] All released fixture stores migrate successfully
- [ ] Migration test suite passes
- [ ] If releasing: new fixture store generated and committed
- [ ] If releasing: registry entry added
- [ ] If squashing unreleased stages: documented and tested
```

## Release Workflow

### Cutting A Release

1. Ensure the migration chain is complete in `PersistenceMigrationPlan`.
2. Run the full migration test suite against all existing fixture stores.
3. Consider squashing unreleased intermediate schema versions (optional, see squashing rules).
4. Generate the fixture store for the current schema version.
5. Add the registry entry to `ReleaseMigrationRegistry`.
6. Commit the fixture store and registry update.
7. Run the full migration test suite again (now including the new fixture).
8. Cut the annotated tag: `git tag -a release/X.Y.Z -m "Release X.Y.Z — SchemaVN"`.
9. Push the tag: `git push origin release/X.Y.Z`.

### After A Release

The tagged schema version and all prior stages are now frozen. Any future schema change must add a new version, never modify existing ones.

## Common Mistakes

### 1. Forgetting to generate the fixture store before tagging

The tag is cut, but no fixture exists. Future migration tests cannot prove upgrade from this release.

Prevention: The release checklist requires fixture generation before tagging.

### 2. Squashing stages that were already released

A released schema version is removed from the chain. Users on that version can no longer upgrade.

Prevention: The squashing rules explicitly forbid this. Guard tests verify every registry entry's schema version appears in the migration chain.

### 3. Editing a fixture store after commit

The fixture no longer represents what actually shipped. Migration tests lie.

Prevention: Fixture stores are treated as immutable binary artifacts. Code review should flag any modification to existing fixture files.

### 4. Seeding fixtures with trivial data

Fixture only contains empty/default/nil values. Migration appears to work but would fail with real user data.

Prevention: Seeded data requirements mandate non-default values, populated optionals, and edge cases.

### 5. Not testing the full chain

Tests only check the last release to current, not older releases.

Prevention: The test suite iterates over every entry in the registry. All releases are tested on every PR.
